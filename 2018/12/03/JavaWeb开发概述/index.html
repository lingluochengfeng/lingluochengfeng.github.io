JavaWeb开发概述

Java Web开发概述

Web相关概念

  web程序程序也就是一般所说的网站，由服务器、客户端浏览器以及网络组成。Web程序的好处是：使用简单，不需要安装、学习，一台电脑，一根网线就可以使用。

Web程序的相关概念

桌面程序（Desktop Program）

  桌面程序（Desktop Program）也成为胖客户端程序（Rich Client Program，简称RCP），必须要装在桌面上才能运行，计算机安装的任何程序都是RCP。比如：办公软件（Word，Excel），聊天工具（QQ，微信）等。

Web程序

  Web程序也成为瘦客户端程序（Thin Client Program，简称TCP），其特点是不需要在客户端安装便能使用，只要计算机能上网就行。

B/S与C/S结构

  按照是否需要访问网络，程序可分为网络程序与非网络程序。其中，网络程序又可以分为B/S结构与C/S结构。

  C/S结构指客户端(Client)/服务器(Server)模式。这种模式需要安装一个桌面程序，桌面程序负责与服务器进行数据交换。一般的网络程序都是C/S结构。
  缺点：需要安装客户端程序。当应用程序升级时，客户端同样需要下载升级程序才能使用新的功能。这样会给客户端带来一定的麻烦，限制了该应用程序的广泛使用，维护麻烦。
  优点：交互性好，服务器压力小，用户体验好，对信息安全控制较强，应用服务器运行数据负荷较轻，部分计算功能在客户端完成

  B/S结构指浏览器(Browser)/服务器(Server)模式。一般网站都是B/S结构的。比如：Google、Baidu。
Web应用程序的访问不需要安装客户端程序，可以通过任一款浏览器来访问各类Web应用程序。
缺点：动画效果受浏览器限制，对信息安装控制较差，应用服务器运行数据负荷较重。大部分计算在服务器端，增加服务器压力。

Web开发背景知识

Web资源分为静态资源和动态资源：
静态资源：指web页面中供人们浏览的数据始终不变。比如：HTML、CSS、JS、图片、多媒体。
动态资源：指web页面中供人们浏览的数据由程序产生，不同时间点访问web页面看到的内容各不相同。比如：JSP/Servlet、ASP、PHP。

URL请求路径

  URL(Uniform Resource Locator)，统一资源定位符是对互联网上资源位置的一种表示，互联网上的每个文件都有一个唯一的URL。
完整格式如下
协议://用户名:密码@域名:端口号/资源位置?参数=值#标志
协议:http、https、ftp等
用户名：密码，常用于ftp访问，路径直接编写账号(一般不写)
域名:域名或IP 地址，都可以访问WEB资源
端口号:程序必须使用端口号， 才可以让另一个计算机访问。http协议的默认端:80
资源位置:用于描述WEB资源在服务器上的位置。
参数＝值:浏览器和服务器交互传递的数据
＃标志:描点， 用于指定页面的某一个位置。

Web通信（Web访问的基本原理）

  WEB采用B/S通信模式，通过超文本传输协议(HTTP,Hypertext transport protocol)进行通信。通过浏览器地址栏编写URL，想服务器发送一个请求，服务器根据请求进行相应的处理，处理完成之后向浏览器做出一个响应，将服务器端资源发送给浏览器。

（1）用户打开浏览器，输入网址，这个地址告诉浏览器要访问互联网中的哪个主机。
（2）浏览器寻找到指定的主机之后，向Web服务器发出请求（request）。
（3）Web服务器接受请求并作出相应的处理，生成处理结果，大多生成HTML格式，也有其他相应的格式。
（4）服务器把相应的结果返回发送给浏览器。
（5）浏览器接收到对应的相应结果后，在浏览器中显示响应结果，比如Web页面。

HTTP超文本传输协议

  浏览器与Web服务器之间沟通的桥梁：HTTP超文本传输协议。
  客户端连上web服务器之后，若想获得web服务器中的某个web资源，需要遵守一定的通讯格式，HTTP协议用于定义客户端与web服务器的通讯格式。

  HTTP协议是浏览器和服务器之间的应用层通信协议。它是基于TCP/IP协议之上的协议，默认端口是80，是一种基于请求-响应模式的协议，不仅保证正确传输超文本文档，还能确定传输文档中的哪一部分，以及哪一部分内容首先显示（如文本先还是图形）等。

  在WWW中，“客户”与“服务”是一个相对的概念，只存在于一个特定的连接期间，即在某个连接中的客户在另一个连接中可能作为服务器。WWW服务器运行时，一直在TCP的80端口（WWW的默认端口）监听，等待着连接出现。

  HTTP（hypertext transfer protocol，超文本传输协议），用于定义WEB浏览器与WEB服务器之间交换数据的过程。

HTTP协议的版本

  • HTTP/1.0，链接后，发送请求，服务器做出相应，链接立即断开，只能获得一个web资源
  • HTTP/1.1，链接后，发送请求，服务器做出响应，链接不会立即断开。再次发送请求，直至一段时间没有操作，自动断开。
  • HTTP/2.0，好像用的不多

  HTTP协议的客户/服务器模式的信息交换过程分为4各过程：建立连接、发送请求、发送响应信息、关闭连接。
  1.建立连接：连接的建立是通过申请套接字(Socket)实现的。客户打开一个套接字并把它约束在一个端口上，如果成功，相当于建立了一个虚拟文件。以后就可以在这个虚拟文件上写数据并通过网络向外传送。
  2.发送请求：打开一个连接后，客户机把请求消息发送到服务器的停留端口上，完成提出请求动作。

Web服务器

常用的服务器：
  1. IIS是微软提供的一种Web服务器，提供对ASP语言的良好支持，通过插件的安装，也可以提供对PHP语言的支持
  2. Apache服务器由Apache基金组织提供的一种Web服务器，其特长是处理静态页面，对于静态页面的处理非常高效。
  3. Tomcat也是Apache基金组织提供的一种Web服务器，提供对JSP和Servlet的支持，通过插件的安装，可以提供对PHP语言的支持，但Tomcat只是一个轻量级的Java Web容器，像EJB这样的服务在Tomcat中无法运行。
  4. JBoss是一个开源的重量级Java Web服务器，在JBoss中，提供对J2EE各种规范的良好支持，而且JBoss通过了Sun公司的J2EE认证，是Sun公司认可的J2EE容器。
  5. 另外J2EE服务器还有BEA的Weblogic和IBM的WebSphere等，适合大型的商业应用。

Web开发技术发展历史

这里简单提下Servlet

Java Servlet：改进的CGI（Common GateWay Interface）

  Servlet为了解决CGI的缺点，即只需启动一个操作系统进程以及一个JVM映像。
  另外，Java可以在运行的时候动态地进行加载，所以可以利用这样的功能加载新的Java代码来处理新的请求。这样就能只启动一次服务器进程，而且只需要加载一次JVM，之后这个JVM再加载另外的类。
  相对于CGI，Servlet有如下几个优点：
  1. 只需要启动一个操作系统进程以及加载一个JVM，大大降低了系统开销。
  2. 如果多个请求需要做同样的处理的时候，只需要加载一个类，大大降低了系统开销。
  3. 所有动态加载的类可以实现对网络协议以及请求解码的代码共享，大大降低了工作量。
  4. Servlet能够直接和Web服务器交互，而普通的CGI不能。Servlet还能在各个程序之间共享数据，使得数据库连接池之类的功能很容易实现。
  虽然Servlet改善了CGI的缺点，但还存在一些缺点：
  Servlet可以建立动态生成的网页，而网页中可以包含从服务器端的Java对象所获得的数据。但Servlet生成网页的方法是在Java类中嵌套HTML标签和表达式。也就是说HTML做一个小小的改动时，都需要修改和重新编译Servlet源文件，然后重新部署到Servlet容器中。或许设计HTML页面和编写Servlet不是同一个人，这就使得Servlet变得很麻烦。

JSP：Servlet模板

  Java Server Pages（JSP）是一种实现普通静态HTML和动态HTML混合编码的技术，它是Servlet API的一个扩展。
  JSP网页可以非常容易地与静态模板结合，包括HTML或XML（Extensible Markup Language）片段，以及生成动态内容的代码。

JSP基本原理

Web容器处理JSP文件请求需要经过三个阶段：
  翻译阶段（Translation phase）：编写好的JSP文件首先会被Web容器中的JSP引擎转换成Java代码，也就是通过Servlet引擎。
  编译阶段（Compilation phase）：JSP文件所翻译成的Java源代码会被编译成可执行的字节码。（可执行的字节码是二进制格式）
  请求阶段（Request phase）：当容器接受了客户端的请求之后，就执行前面已经编译成二进制字节码的JSP文件。处理完请求之后，容器再把生成的页面反馈给客户端进行展示。
  如下图所示：
  注意：一旦容器把JSP文件编译和翻译之后，来自客户端的每一个JSP请求就可以重用这个编译好的二进制字节码，没有必要再把同一个JSP文件进行编译和翻译，这大大提高了Web应用系统的性能。如果对JSP进行了修改，容器会及时地探测到这个修改，并进行重新翻译和编辑。所以JSP文件在第一次请求时会比较慢，而之后同样JSP文件的请求会非常快。
